# Dev Cycle Orchestrator — Linear-driven spec-first workflow

## Problem Frame

Разработка фичей сейчас не имеет единого структурированного цикла. Задачи в Linear, решения в голове, код в репозитории — но нет связного процесса research → документирование решений → декомпозиция → реализация. Хочется spec-driven development с Linear как источником задач и ADR как центральным артефактом принятия решений.

## Requirements

- R1. **Просмотр задач из Linear** — skill `/my-tasks` показывает задачи назначенные на текущего пользователя (определяется через `viewer` query Linear API по personal token). Фильтрация по проекту/команде задаётся в конфиге проекта (CLAUDE.md или переменная окружения, напр. `LINEAR_PROJECT`). Пользователь выбирает задачу для работы.

- R2. **Research-фаза** — skill `/research` для выбранной задачи: собирает контекст из описания Linear-задачи, исследует кодовую базу, внешние источники, формулирует варианты решений. Результат — черновик ADR в `docs/adr/` со статусом `proposed` (отдельного research-артефакта нет, research материализуется в секции Context ADR).

- R3. **ADR как центральный артефакт** — skill `/write-adr` создаёт Architecture Decision Record в `docs/adr/` на основе research. Формат — классический MADR: Status, Context, Decision, Consequences. Фронтматтер: дата, статус (`proposed`/`accepted`/`superseded`), ссылка на Linear issue.

- R4. **Декомпозиция на задачи** — из принятого ADR генерируются подзадачи. Подзадачи создаются как sub-issues в Linear (привязаны к родительской задаче). Каждая подзадача содержит достаточно контекста для автономной работы.

- R5. **Маркировка параллелизуемости** — при декомпозиции задачи размечаются: какие можно делать параллельно (независимые), какие последовательно (зависимые). Это влияет на то, как `/dev-cycle` предлагает их выполнять.

- R6. **Выполнение работы** — запуск ce-plan → ce-work для каждой подзадачи. Параллельные задачи могут запускаться через dispatching-parallel-agents. Статус в Linear обновляется после подтверждения на чекпоинте (пользователь подтверждает завершение → статус меняется на Done).

- R7. **Оркестратор `/dev-cycle`** — единая точка входа, которая определяет текущий этап и предлагает следующий шаг:
  - Нет задачи → `/my-tasks`
  - Задача выбрана → классификация сложности:
    - **Лёгкий путь** (баг-фикс, мелкая задача) → сразу ce-plan → ce-work, без ADR
    - **Полный путь** (фича, архитектурное решение) → research → ADR → декомпозиция → работа
  - Пользователь выбирает путь на чекпоинте (оркестратор может предложить рекомендацию)
  - Research есть, нет ADR → `/write-adr`
  - ADR принят, нет подзадач → декомпозиция
  - Подзадачи есть → показать прогресс, предложить начать/продолжить работу

- R8. **Чекпоинты** — после каждого этапа (research, ADR, декомпозиция, каждая подзадача) — пауза для ревью и подтверждения пользователем.

- R9. **Отдельные входные точки** — каждый этап доступен как самостоятельная команда (`/my-tasks`, `/research`, `/write-adr`), не только через оркестратор.

## Success Criteria

- Пользователь может от Linear-задачи дойти до рабочего кода через структурированный цикл (полный с ADR или лёгкий без)
- Решения задокументированы в `docs/adr/` до начала кодирования
- Подзадачи в Linear отражают реальную декомпозицию из ADR
- Независимые задачи параллелятся, зависимые — последовательно
- Весь цикл управляем через `/dev-cycle` или поэтапно через отдельные команды

## Scope Boundaries

- **Вне скоупа**: публикация ADR в Outline (ручной процесс)
- **Вне скоупа**: CI/CD интеграция, автоматический деплой
- **Вне скоупа**: мультикомандная работа (workflow для одного разработчика)
- **Вне скоупа**: конфигурируемость чекпоинтов (всегда human-in-the-loop на каждом шаге)

## Key Decisions

- **Идентификация пользователя**: Linear API personal token → `viewer` query (не git config, не конфиг)
- **ADR формат**: классический MADR (Status, Context, Decision, Consequences)
- **ADR хранение**: `docs/adr/` в репозитории, коммитится в git
- **Фильтрация задач**: по проекту/команде через конфиг (`LINEAR_PROJECT` или CLAUDE.md)
- **Оркестрация**: поверх существующих ce-* скиллов (ce-brainstorm, ce-plan, ce-work), не замена им
- **Архитектура**: один оркестратор `/dev-cycle` + отдельные команды для каждого этапа

## Dependencies / Assumptions

- Linear API personal token доступен через переменную окружения (напр. `LINEAR_API_KEY`)
- Linear GraphQL API доступен и стабилен
- ce-* скиллы (brainstorm, plan, work) вызываемы из других скиллов
- Проект использует git для version control
